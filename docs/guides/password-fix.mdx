## Issue Overview

Users encountered a registration error: **"Password is too long. Maximum is 72 bytes"** even with passwords as short as 12 characters.

<Warning>
  This issue was caused by incompatibility between bcrypt versions and passlib 1.7.4
</Warning>

## Root Cause Analysis

### Timeline of Issues

<Steps>
  <Step title="Initial Problem (bcrypt 5.0.0)">
    - bcrypt 5.0.0 introduced breaking API changes
    - passlib 1.7.4 doesn't support these changes
    - All passwords were incorrectly rejected
  </Step>
  
  <Step title="Second Attempt (bcrypt 4.1.3)">
    - bcrypt 4.1.3 removed the `__about__` attribute
    - passlib 1.7.4 tries to read `_bcrypt.__about__.__version__`
    - Caused AttributeError and fallback to incorrect behavior
  </Step>
  
  <Step title="Final Solution (bcrypt 3.2.2)">
    - bcrypt 3.2.2 is fully compatible with passlib 1.7.4
    - All password operations work correctly
    - This is the last stable version before breaking changes
  </Step>
</Steps>

## The Solution

### Version Compatibility

| Package | Version | Status |
|---------|---------|--------|
| passlib | 1.7.4 | ✅ Stable (2020) |
| bcrypt | 3.2.2 | ✅ Compatible |
| bcrypt | 4.x | ❌ Incompatible (`__about__` removed) |
| bcrypt | 5.x | ❌ Incompatible (API changes) |

### Changes Made

<CodeGroup>
```python requirements.txt
# Authentication & Security

```

```python vmledger/services/auth_service.py
def _validate_password_complexity(self, password: str) -> None:
    """Validate password meets complexity requirements."""
    # Check bcrypt byte limit (72 bytes)
    password_bytes = len(password.encode('utf-8'))
    if password_bytes > 72:
        raise PasswordComplexityError(
            f"Password is too long ({password_bytes} bytes). Maximum is 72 bytes."
        )
    # ... other validations
```

```typescript frontend/lib/validation.ts
export function isValidPassword(password: string): boolean {
  if (password.length < 12) return false;
  
  // Check bcrypt byte limit (72 bytes)
```

## Password Requirements

<Note>
  Passwords must meet these requirements to be accepted:
</Note>

| Requirement | Details |
|-------------|---------|
| **Minimum Length** | 12 characters |
| **Maximum Length** | 72 bytes (≈72 ASCII characters) |
| **Uppercase** | At least one (A-Z) |
| **Lowercase** | At least one (a-z) |
| **Number** | At least one (0-9) |
| **Special Character** | At least one (!@#$%^&*(),.?":{}|<>) |

### Understanding Byte Limits

<Accordion title="Why 72 bytes?">
  Bcrypt has a built-in limitation of 72 bytes. This is a security feature, not a bug:
  
  - **Security**: Longer passwords don't increase security beyond a certain point
  - **Performance**: Bcrypt is intentionally slow to prevent brute-force attacks
  - **Standard**: 72 bytes is the bcrypt specification limit
</Accordion>

<Accordion title="ASCII vs Unicode">
  - **ASCII characters** (a-z, A-Z, 0-9, common symbols): 1 byte each
  - **Unicode characters** (emojis, accented letters): 2-4 bytes each
  
  Example:
  - `MyPassword123!` = 14 bytes ✅
  - `MyPässwörd123!` = 16 bytes (ä and ö are 2 bytes each) ✅
  - `MyPassword🔒123!` = 18 bytes (🔒 is 4 bytes) ✅
</Accordion>

## Technical Details

### Why bcrypt 3.2.2?

<Info>
  passlib 1.7.4 was released in 2018 and expects the older bcrypt API
</Info>

- **bcrypt 3.2.2** (2021): Last version with full passlib compatibility
- **bcrypt 4.x** (2022): Removed `__about__` attribute
- **bcrypt 5.x** (2024): Additional breaking API changes

### Alternative Solutions (Not Chosen)

<AccordionGroup>
  <Accordion title="Upgrade passlib">
    **Why not?** passlib hasn't been updated since 2020, no newer version available
  </Accordion>
  
  <Accordion title="Switch to argon2">
    **Why not?** Would require rewriting all authentication code and migrating existing password hashes
  </Accordion>
  
  <Accordion title="Use bcrypt directly">
    **Why not?** Would lose passlib's password context features (algorithm migration, hash verification, etc.)
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Still getting 72-byte error">
    1. Verify bcrypt version: `docker exec vmledger-api pip show bcrypt`
    2. Restart API container: `docker restart vmledger-api`
    3. Check password byte length: Use online UTF-8 byte counter
    4. Try a simple ASCII password first: `TestPass123!`
  </Accordion>
  
  <Accordion title="AttributeError: __about__">
    This means bcrypt 4.x or 5.x is still installed:
    ```bash
    docker exec vmledger-api pip install "bcrypt==3.2.2" --force-reinstall
    docker restart vmledger-api
    ```
  </Accordion>
  
  <Accordion title="Frontend still shows error">
    1. Clear browser cache
    2. Hard refresh (Ctrl+Shift+R or Cmd+Shift+R)
    3. Check frontend validation in `frontend/lib/validation.ts`
  </Accordion>
</AccordionGroup>
